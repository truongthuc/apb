# APBF Project Rules

## Language Policy

- Communication with the project owner may be in Vietnamese when requested.
- All repository artifacts must be written in English.
- Formal project rules, specifications, documentation, templates, configuration notes, and implementation-facing materials must use English.
- Code identifiers, file names, configuration keys, commands, and technical terms must remain in English.
- Do not mix languages inside repository artifacts unless quoting source material or preserving an explicit external requirement.

## Boundary Policy

- APBF is not an application.
- APBF is not an AI Orchestrator.
- Do not introduce AI Orchestrator domain logic into APBF.
- Do not add application-specific business rules, runtime flows, agents, tools, prompts, or orchestration behavior to APBF.
- Keep APBF focused on bootstrap framework structure, conventions, and project-starting guardrails.
- Stay within the requested repository scope and do not create files outside the requested path.
- Do not create planning materials at this stage.
- Do not create templates at this stage.
- Do not create documentation at this stage.
- Do not write code at this stage.

## Repository Role

- APBF defines foundational rules and boundaries for initializing AI-related projects.
- APBF provides repository-level guidance for future project setup without becoming the project being set up.
- APBF should remain domain-neutral, implementation-neutral, and reusable.
- APBF may later contain framework conventions, templates, or documentation only after explicit approval.
- The current repository initialization step is limited to establishing baseline agent rules.

## Development Workflow

- Follow explicit user instructions before expanding repository structure.
- Make the smallest change that satisfies the current request.
- Create or modify only files that are explicitly requested.
- Confirm repository boundaries before adding planning, templates, documentation, or code.
- Preserve user-owned changes and avoid overwriting unrelated work.
- After each initialization step, summarize the change and wait for review before continuing.
